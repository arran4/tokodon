Is 100 really missing?!
Let's check `hasFromId` logic in `fillTimeline`:
```cpp
    const bool hasFromId = !fromId.isEmpty() && !query.hasQueryItem(QStringLiteral("max_id"));
    if (hasFromId) {
        query.addQueryItem(QStringLiteral("max_id"), fromId);
    }
```
If we use `max_id = 100`, the API returns posts strictly OLDER than 100. (i.e. 99, 98).
Then we use `m_prev`.
`m_prev` was returned by the API when we fetched `max_id = 100`.
The link header `prev` is generated by the server.
For `max_id=100`, the server returns 99, 98.
The server generates `prev` link with `min_id=99`.
When Tokodon calls `fillTimeline({}, true)`, it fetches `m_prev`, which is `min_id=99`.
The API returns posts strictly NEWER than 99.
Which means it returns 102, 101, 100!
Because 100 is newer than 99!
So 100 IS included in the newer posts!
Wait, but if 100 is returned, and 99 is already there, then the timeline will have 102, 101, 100, 99, 98.
And `pos` (the number of newer posts) is 3 (for 102, 101, 100).
So `repositionAt(pos)` will position at index 3, which is 99!
If we position at 99, the user is looking at the first post they have ALREADY READ.
Is that the intended behavior?
"Continue reading where you last left off"
If they see 99, they see what they already read. To see unread posts, they scroll up.
This is exactly how many timeline viewers work! They show the read marker above the post you just read, so you see the last read post to get context, and then scroll up!
Wait. If they position at 99, then 100 is at index 2 (above the viewport).
If the UI has a read marker BETWEEN 100 and 99, it is drawn above 99.
If 99 is at the top of the viewport, the read marker is AT THE TOP of the viewport.
So the user sees the read marker and scrolls up to see 100, 101, 102.
This is correct UI behavior!

So the ONLY bug is that `findLatestPostId` is completely broken, causing it to overwrite `m_lastReadId` with 99, or with older posts when scrolling!
Because when it fetches the older posts (99, 98), `m_timeline` DOES NOT have 100 yet!
So QML draws 99, calls `updateReadMarker(99)`.
`findLatestPostId({99, 100})` searches `m_timeline`.
`m_timeline` only has 99, 98. It DOES NOT have 100 yet (because the second fetch hasn't completed).
So `findLatestPostId` returns 99!
And Tokodon overwrites `m_lastReadId` to 99!
Then the second fetch completes, prepending 100.
But it's too late! `m_lastReadId` was already changed to 99!
And sent to the server!
So the server now thinks the last read ID is 99!
Next time the user opens the app, it loads 99!
Then it fetches older posts (98, 97).
QML draws 98, calls `updateReadMarker(98)`.
`findLatestPostId({98, 99})` returns 98.
`m_lastReadId` is overwritten to 98!
So EVERY TIME the user opens the app, the read marker SLIPS DOWN BY ONE POST!
This is exactly why it "doesn't remember my last scroll position"!!!
Because it constantly loses it and moves backward!

To fix this, we simply need to fix `findLatestPostId` to do direct string comparison (with length check) instead of searching the timeline!
